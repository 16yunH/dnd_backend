1. 核心需求剖析智能建卡：用户自然语言输入要求，AI 结合规则生成标准化的角色卡数据。AI DM（主持人）：根据玩家奇思妙想，在 D&D 规则框架和骰子判定结果下，推演合理的剧情发展。
2. 成本控制与 Token 优化方案模型选型：推荐使用极低成本/免费的云端 API（如 DeepSeek-V3/R1、智谱 GLM-4-Flash、Gemini 1.5 Flash），或使用本地算力部署（Ollama + 独显）。逻辑剥离：绝对不要让 AI 计算随机骰子。由后端程序生成随机数（如 1d20 = 15），加上角色属性修正后，将“最终结果”以文本形式喂给 AI 生成剧情，大幅节省 Token。上下文压缩：对于超长跑团记录，使用小模型定期生成“剧情摘要”，用摘要替换早期对话历史。RAG（检索增强生成）：不把整本规则书塞入提示词。根据玩家当前动作，动态检索相关规则给 AI 看。
3. 规则与数据的格式化角色卡/物品/怪物属性：统一整理为 JSON 格式。方便前端 UI 渲染，方便后端数值计算，也完美契合现代 AI 的“结构化输出（Function Calling）”能力。纯文字规则/剧情背景：统一整理为 Markdown (.md) 格式。AI 理解层级最好。现有 .chm 规则书处理：使用终端命令 (hh.exe -decompile) 反编译为 HTML。使用脚本（如 Python 的 markdownify）清洗并转为 Markdown。按战斗、魔法、职业等切片分类。
4. 数据库与存储架构（极致性价比路线）针对你的复杂需求（既有规则检索，又有用户数据存档），强烈推荐 PostgreSQL + pgvector 插件（或基于它的免费平台如 Supabase）。该方案可以“一库多用”：关系型存储：存用户账号、跑团房间号。JSON 存储：存千变万化的角色卡属性和背包物品（PostgreSQL 原生支撑 JSONB）。向量存储 (Vector)：把切片好的 Markdown 规则转化为向量存入，供 AI 随时检索（RAG）。聊天记录存档：存玩家和 DM 的历史对话及总结摘要。图片/地图资源：存放在独立的对象存储（OSS/S3）或本地静态目录下，数据库只存 URL。
5. 大致分工应该为：规则整理：角色卡规则，战役规则->用5e不全书前端:包括创建角色卡和具体游戏的前端UI,游玩过程以文字形式描述，需要一副地图标定各个角色的位置；后端：主要是存储和读取调度，不同职业规则，不同战役规则，角色卡数据库等的分类索引。
